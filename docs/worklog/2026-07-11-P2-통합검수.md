## 2. 배치 경계 시나리오 (임시 스택 실측 — curl + 웹 SSR + pdftotext)

### (A) 팀 전 과정: 팀(P2a) → 서명(P2b 인프라) → 파생 → 소득 리포트(P2d) 정합
실데이터 시드 후 라이브 HTTP 로 재현. **수동 계산 대조 일치**:
- 반장(박통합반장): 팀 확인서 420,000(홍통합 1.5공수×180,000=270,000 + 김수기 1공수×150,000) + 단독 확인서 200,000 = **총 청구 620,000**.
  - `teamPayout` = 270,000 + 150,000 = **420,000**(가입 팀원 + 수기 미가입 팀원 모두 포함).
  - `netBilled` = 620,000 − 420,000 = **200,000**(= 반장 단독 작업분, 팀 내 본인 몫 0). ✓ 정확.
  - 공수 2.5(팀), 일수 2. **소득 리포트 PDF**(pdftotext)도 동일 값 렌더(총 청구 620,000 / 팀 지급분 420,000 / 순소득 참고 200,000 / 2.5공수).
- 팀원(홍통합): 파생 항목만 → **총 청구 270,000, teamPayout 0, 공수 1.5, 상대=반장명**. ✓ 이중 합산 없음(반장·팀원 profileId 분리, 한 리포트 내 공존 안 함).
- **웹 팀 명단 표 렌더(P2a→P2b 이월)**: `/c/{token}` SSR HTML 에 `팀 명단`·홍통합·김수기·개별 금액 270,000/150,000·`팀 작업 합계` 420,000 확인. 배치 경계 웹 산출물 라이브 동작.

### (B) 계약서(P2b) + TBM(P2c) 결합 → 안전 리포트
- 사업장이 연결 작업자와 표준근로계약서 작성→사업주 서명→전송→**작업자 앱내 서명 SIGNED** 왕복 성공.
- 같은 작업자가 TBM 참석(가입 참석자 + 수기 참석자 2명) → **ack 성공**, **재ack 409**(원자적 최초 1회 전이).
- **안전 리포트 PDF**(pdftotext): `■ TBM(안전점검회의) / 2026-07-11 강동 현장 지하1층 / 중장비 협착·충돌·폭염 온열질환 / 참석·확인 2/1` + `■ 발송·확인 기록` 에 TBM 기록(작성자) + TBM 확인(참석자, 확인시각) 각각 append. safety_logs 정합 확인. 이름 마스킹(안*주/안***자) 일관 적용.

### (C) 웹 계약서(P2b) 라이브 렌더
- `/lc/{token}` ko: 표준근로계약서·근로계약기간·강동 현장·형틀목공·170,000·정본 안내(정본/한국어)·사업주/근로자 서명 렌더.
- `/lc/{token}?lang=vi`: 조항 라벨 vi 번역(Hợp đồng lao động·Người sử dụng·tiếng Hàn 정본 안내), **데이터 값은 원문 유지**(강동 현장·형틀목공). 설계(조항 번역/값 원문) 대로 동작.

### 공수 일관성
- 소득 리포트 `gongsuOf`: 일반 GONGSU(기본항목 수량)·팀 확인서(팀원 공수 합)·팀 파생(본인 몫 공수) 3경로 모두 반영. 라이브 실측(반장 2.5·팀원 1.5)로 확인. `entriesForRange` 는 월간(`entriesForMonth`)과 동일 기준(작업일 우선·파생 원확인서 작업일·수기 생성일)이라 월/연 집계 정합.

## 3. 정책 일관성 판정 (P2d 제기 3건 — 권고)

1. **teamPayout 정의(수기 미가입 포함) 적절성 → 현행 유지 권고.**
   반장 관점에서 수기(미가입) 팀원 지급분도 실제 현금 유출이므로, 포함해야 `netBilled`(순소득 참고)가 정확해진다.
   제외 시 순소득이 과대계상됨. 라벨("팀 지급분(팀원 몫)")도 정직. 반장 본인 몫(profileId=본인)은 필터로 제외되어 netBilled 에 남음 — 일관.

2. **소득 리포트의 세금계산서 발행분(taxInvoicedAt) 미구분 → 문제 아님(현행 유지).**
   소득 리포트(종소세 소득 집계)와 P1 세금계산서 데이터(부가세 세금계산서 발행 대상)는 **서로 다른 질문**에 답한다.
   종합소득세 소득은 세금계산서 발행 여부와 무관하게 전액 집계하는 것이 옳다. 두 기능을 섞으면 혼동. 분리 유지.
   (선택: 앱 UI 에 "세금계산서 발행 여부와 무관한 소득 집계" 한 줄 안내로 사용자 혼동 예방 가능.)

3. **P2b 백로그(계약서 revoke 미노출 / 사업장 웹 계약서 UI 없음) → 출시 차단 아님.**
   - **revoke 미노출**: 확인서(`confirmations`)도 revoke 엔드포인트가 없다(무효화 엔드포인트는 서류 묶음 `document-shares` 에만 존재). 계약서·확인서 모두 `revokedAt` 공개열람 차단 가드는 있으나 트리거 엔드포인트만 미구현 — **기존 패턴과 일관**한 기능 공백이지 회귀·결함 아님. → 백로그: 확인서·계약서 공통 revoke 엔드포인트.
   - **사업장 웹 계약서 UI 없음**: 앱에 사업장 계약서 작성/서명 전체 흐름 존재, 웹은 공개 열람·서명(`/lc`)만. 확인서 발행도 앱 우선(웹은 공개 열람) — **앱 우선 사업장 흐름과 일관**. → 백로그.

## 4. 코드 리뷰 스팟 체크

- **권한 격리(타 사업장 접근)**: `labor-contracts.ownedContractOrThrow`(`business.ownerId !== userId → 404`), `tbm.ownedRecordOrThrow`(`r.business.ownerId !== userId → 404`) 모두 소유자 강제. e2e 권한 격리 케이스(남 404·남 사업장 404)로 커버·통과.
- **public 토큰 만료 정책(확인서와 일관?)**: 계약서·확인서 공개 열람 **둘 다 만료 없음, revoke(revokedAt)만 체크** — **일관**. (서류 묶음만 7/14/30일 만료.)
- **TBM 당일 수정 창 시간대(KST)**: `isSameKstDay(a,b) = toKstDateStr(a)===toKstDateStr(b)` — KST 날짜 문자열 비교로 자정 경계 정확. `assertEditable` 로 수정·삭제 모두 당일 강제. e2e(createdAt 과거 조작 후 409) 통과.
- **income-report 대량 데이터 성능**: `entriesForRange` 는 기간(최대 24개월) 전 장부 항목을 3개 include 와 함께 메모리 로드 후 O(n) 집계. 사용자 1인·연간 범위이므로 데이터량 유한(현실적으로 수백 건 이하) — 현 규모에서 허용. N+1 없음. 대규모 사용자·다년 조회로 커지면 DB 측 집계(groupBy)로 최적화 여지(백로그·비차단).
