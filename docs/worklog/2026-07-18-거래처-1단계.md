# 거래처(Partners) 1단계 — 자동 수집 + 문자/전화 바로가기 (backend + app)

일자: 2026-07-18
범위: backend + app (web 무수정). STRATEGY "거래처 (2026-07-18 → 2단계 확정)" 1단계.

## 목표
작업자의 거래처를 확인서·장부의 수기 상대(companyName+contact)에서 **자동 수집**해 목록으로 제공하고,
문자 빠른 보내기 수신인으로 바로 쓸 수 있게 한다. 등록 작업 0 — 수동 등록 화면이 아니라 이미 쌓인
데이터의 뷰 + 선택적 보강(별칭·사업자번호·이메일·메모).

## 설계 결정

### 스키마 — `partners` 테이블 (additive 마이그레이션 1개)
`20260718120000_partners`: CREATE TABLE partners(id, profileId FK cascade, name, phone?, alias?,
bizNumber?, email?, memo?, createdAt, updatedAt) + index(profileId) + **unique(profileId, name)**.

- **식별 키 = (profileId, name), NOT (profileId, name, phone)**. 근거: 장부 by-company·확인서 집계가
  모두 `companyName(=name)` 만으로 그룹핑한다. name 을 자연 키로 삼아야 통계(미수/건수)가 파편화되지
  않는다. (name, phone) 키로 하면 같은 상호의 연락처가 바뀔 때 상대가 쪼개져 미수/건수가 분리된다.
  phone 은 name 하위의 **가변 속성**(문자/전화용)으로 두고 자동 갱신한다. (schema.prisma 주석 명시)
- **통계는 저장하지 않는다**: 확인서 건수·미수 잔액·최근 작업일은 partners 행에 저장하지 않고 매 조회 시
  confirmations/ledger 에서 **파생 계산**한다(정합성·staleness 회피).

### 병합(merge) vs 백필(backfill) — 병합 채택
`GET /partners` 가 **확인서(원천) + 장부(미수) + partners 보강행 + 연결 사업장**을 조회 시 병합해
반환한다. partners 테이블은 "보강 정보 저장소"일 뿐, 목록의 진실은 확인서다.
- 별도 백필 엔드포인트/데이터 마이그레이션 **불필요**. 조회 시 확인서에만 있고 partners 행이 없는 수기
  상대를 **lazy 생성**(createMany skipDuplicates)해 PATCH 대상 id 를 안정적으로 확보. idempotent.
- **근거**: 가장 단순. 원천이 확인서이므로 마이그레이션 시점 데이터 이관이 불필요하고, 기존 확인서도
  첫 GET 에서 자동 수집된다.

### 자동 upsert 훅
확인서 create/update/duplicate 시 수기 상대(businessId 없는 companyName)를 partners 에 upsert
(`PartnersService.safeUpsertFromManualCounterparty`, try/catch — 실패해도 확인서 저장 성공).
연결(businessId) 상대는 businesses 가 원천이라 partner 행을 만들지 않는다.

### 연결(승격) 상대
businessId 있는 확인서 상대도 목록에 포함(문자·전화 용도). 응답에 `businessId`·`linked:true`·사업장명·
소유자 전화 포함, `id:null`(보강 편집 대상 아님).

## API
- `GET /partners` → { count, items[] }. item: id|null, businessId|null, linked, name, phone, alias,
  bizNumber, email, memo, confirmationCount, outstanding, paid, lastWorkedDate. 정렬 최근 작업일 desc.
  (미수는 ledger 파생, **derived=true 팀원 몫 제외**)
- `PATCH /partners/:id` — alias/bizNumber/email/memo 만(ValidationPipe whitelist + DTO). 본인 것만(404).
  빈 문자열 → null(비우기).
- `DELETE /partners/:id` — hard delete(soft 불필요: 확인서 남아 있으면 다음 GET lazy 재수집).
