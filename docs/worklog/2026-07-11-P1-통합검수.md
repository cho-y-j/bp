## 3. 코드 리뷰 스팟 결과

### 3-A. [중요] 웹 외부 열람 — 비한국어에서 공수(GONGSU) 단위 소실
- **현상**(라이브 SSR 실측):
  - 네팔어 `/c?lang=ne`: `आधार ₩१,८०,००० × 1.5 · ₩२,७०,०००` → **"공수" 표기 없음**, 라벨은 일반 "आधार(Base)".
  - 한국어 `/c?lang=ko`: `기본(공수) 180,000원 × 1.5` → 공수 정상.
- **원인**: `web/components/PaperConfirmation.tsx`
  - 수량 렌더(L118~131)가 `it.unit`(='공수')를 **전혀 출력하지 않음** — 모든 언어에서 `rate × quantity` 만.
  - 비한국어 `itemLabel`(L69~73)이 BASE→`amtBase`(일반 "Base") 로 번역해 라벨의 "(공수)"까지 소실.
  - 응답 데이터에는 `unit:'공수'`·`label:'기본(공수)'` 가 있으나 프런트가 버림.
- **영향**: 공수는 P1 헤드라인 기능이자 "일용직 표준 언어"이고, 외부 다국어 열람은 콜드스타트 서명(증거) 서면. PDF·앱·한국어웹 4개 면은 공수를 표기하나 **정확히 태스크가 지목한 배치 경계 면(외국인 웹 열람)만** 누락. 금액(270,000)은 정확해 법적/정산 파손은 아니나 단위 의미가 안 보임.
- **수정 제안**(소규모): 수량 span 에 `it.unit` 있으면 병기(예: `× 1.5공수`, 단위 문자열은 PDF 정책대로 한국어 유지 허용) + 필요 시 GONGSU 기본라벨에 공수 반영.

### 3-B. [정상] 공수 Decimal 반올림 경계
- 175,001 × 0.5 = 87,500.5 → 백엔드 `Math.round`=**87,501**. 앱 `num.round()`도 87,501(양수 half-away). 프리뷰↔서버 일치, ±1 불일치 없음. ✅
- `validateGongsuQuantity`(0.1 단위, ×10 반올림 오차 1e-6) 백엔드·앱 동일 로직. ✅

### 3-C. [사소] 미사용 앱 ARB 키 12개
`download, amtBase, paperGuide, paperMemo, paperSignHead, shareCount, shareValidUntil, shareMasked, statusTransientTitle/Msg, statusNotFoundTitle/Msg` — 웹 사전에서 이식됐으나 앱 코드 미참조(× 6언어 = 번역 낭비). 기능 영향 없음. `paperGuide/paperMemo/paperSignHead` 미사용은 앱 종이카드가 해당 필드를 노출 안 하는 기존 스코프 차이(P1 회귀 아님).

### 3-D. [사소] 초안 큐 동시성
- `DraftQueue.flush()` 는 `_flushing` 가드 + 스냅샷(`[...state]`) 순회 + id 기반 `remove` → **flush 중 enqueue 된 새 초안은 유실 없음**(다음 라운드로 보존). ✅
- 단, 새 초안은 **다음 connectivity 이벤트까지 자동전송 안 됨**(현 flush 스냅샷 밖). 홈 대기배너 + 수동 "지금 모두 전송"으로 커버 → 데이터 손실 아닌 지연 엣지(사소).
- enqueue/remove 의 `_storage.save` 인터리브는 이론적 prefs 마지막쓰기 경합(단일 아이솔레이트 async, 관측 문제 없음).

### 3-E. [정상] 위젯 브리지 민감정보 — 최소화 원칙 부합
- App Group `group.kr.workon` 공유 값: state/brand/오늘 라벨·현장명·시간/미수 라벨·금액/동기화시각/로그인문구. **전화번호·토큰·이름 등 PII 없음**. 현장명+미수금액은 위젯 표시에 필수인 최소치. 로그아웃 시 전 키 빈값 덮어써 잔상 제거. ✅

### 3-F. [정상] 카카오 link 멱등성
