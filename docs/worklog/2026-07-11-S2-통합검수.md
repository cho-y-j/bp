# S2 통합 검수 — 수정 작업 기록 (2026-07-11)

> 독립 검수 에이전트가 지적한 결함(중요 1 · 사소 6)을 수정하고 회귀 검증한 기록.
> 테스트 DB: 임시 postgres(:5433, 컨테이너 `jakeobon-testpg`). live-db(:5432)는 사용 금지 준수.

## 검수 요약

### [중요] 확인서 상대 businessId 연결 검증 부재
- 문제: `resolveCounterparty` 가 businessId 존재만 확인하고, 요청자(작업자)와 그 사업장 간
  ACCEPTED 연결을 검증하지 않았다. → **제3자가 임의 businessId 로 확인서를 작성**해 타 사업장을
  상대로 지정할 수 있었다(정산·수신함 오염 가능). bizSign 도 상태 무관하게 서명 가능했다.

### [사소] 6건
1. viewLogs 무제한 누적(확인서·서류공유 public 열람 로그) — 메모리/행 비대화
2. 서명 TOCTOU — 상태 조회 후 update 사이 경합으로 이중 서명 가능성
3. 정산 pay — read-modify-write 루프가 비트랜잭션(부분 실패 시 불일치)
4. job 상태전이 미강제 — start/complete 가 현재 상태를 검증하지 않음
5. 승격 전역 스캔(manualContact) 인덱스 부재
6. job 사진 업로드 MIME 미검증(임의 파일 저장 가능)

## 수정 결과

### [중요]
- `confirmations.service.ts`
  - `assertConnectedToBusiness(userId, businessId)` 추가 — Connection 이 ACCEPTED 아니면
    `400 NOT_CONNECTED`("미연결 상대는 수기 회사명으로 작성"). jobs.service 의 작업지시 연결
    검증과 동일 기준으로 통일.
  - 적용: `resolveCounterparty`(create), `duplicate`(원본 businessId), `update`(PATCH, businessId 유지본).
  - `bizSign`: 소유자 검증에 더해 **상태 SENT 인 경우만** 서명 가능(`NOT_SIGNABLE` 409). 비대상은
    기존대로 403(FORBIDDEN)/404(CONFIRMATION_NOT_FOUND).

### [사소]
1. **viewLogs cap 50 + viewCount**: 확인서/서류공유 publicView 에서 `slice(-50)` 로 최근 50개만
   유지, 누적 총계는 `viewCount Int @default(0)` 카운터(`{ increment: 1 }`)로. 공유 목록의 viewCount
   는 카운터를 반환하도록 변경. 마이그레이션 `20260711050000_s2_review_hardening`.
2. **서명 TOCTOU**: `applySignature` 를 `updateMany where status != SIGNED` 원자적 전이로 교체.
   count=0 이면 `409 ALREADY_SIGNED`. 이미지 검증(400)은 상태 전이 전에 수행.
3. **정산 pay 트랜잭션**: 조회+검증+입금 update 루프를 `prisma.$transaction` 으로 감쌈(알림은 커밋 후).
4. **job 상태전이 강제**: `start` 는 `acceptedAt` 있는 `SCHEDULED` 만(아니면 `409 JOB_NOT_STARTABLE`),
   `complete` 는 `IN_PROGRESS` 만(아니면 `409 JOB_NOT_COMPLETABLE`).
5. **승격 스캔 인덱스**: `Confirmation.manualContact`, `Job.manualContact` `@@index` 추가(위 마이그레이션 동봉).
6. **job 사진 MIME**: jpg/png/heic/webp(+heif) 만 허용, 그 외 `400 UNSUPPORTED_PHOTO_TYPE`(documents 업로드 기준과 정합).

## 남긴 백로그
- **승격 전역 스캔 최적화**: `PromotionService.promoteForBusiness` 는 여전히
  `confirmation/job` 전체에서 `businessId=null AND manualContact != null` 을 스캔한다. 이번에
  `manualContact` 인덱스를 추가해 정확 매칭 후보 축소는 되지만, 전화번호 정규화 매칭이 앱단
  (`selectPromotable`)에서 이뤄져 여전히 full 후보 로드가 필요하다. 데이터 증가 시
  정규화 전화 컬럼(예: `manualContactNorm`) 비정규화 + 해당 컬럼 인덱스로 DB 측 필터링 전환 권장.
