# 리프레시 토큰 기반 자동 로그인 연장 (backend + app)

작성일: 2026-07-17
범위: backend + app (web/ 무수정 — 타 에이전트 작업 중)
포트 규칙 준수: 임시 pg **5438**, 백엔드 **3060** 만 사용. live-db(5432)·사용자 테스트 스택(5439/3010/3001)·타 에이전트(5436/3040)·기타(5435) 무접촉.

## 목표
기존은 JWT 7일 단일 토큰(만료 시 재로그인). 액세스 토큰을 30분으로 단축하고,
불투명 리프레시 토큰(슬라이딩 180일)을 도입해 **앱을 쓰는 한 재로그인이 없도록** 자동 연장한다.
회전(rotation)·재사용 감지(탈취 방어)·기기 상한·정리 크론을 포함한다.

## 무엇을 / 어떻게 했는가

### 백엔드 (NestJS + Prisma)
1. **스키마(마이그레이션 1개)** — `20260717120000_refresh_tokens`
   - `refresh_tokens`: `profileId`(FK, onDelete Cascade), `tokenHash`(sha256 hex, @unique), `expiresAt`(슬라이딩 180일), `lastUsedAt?`, `deviceId?`(선택), `revokedAt?`, createdAt/updatedAt.
   - 인덱스: `profileId`, `tokenHash`, `expiresAt`. Profile 에 `refreshTokens RefreshToken[]` 관계 추가.
2. **`RefreshTokenService`** (`src/auth/refresh-token.service.ts`)
   - 토큰: `crypto.randomBytes(32).toString('hex')` → 불투명 랜덤 **64자**. 저장은 **sha256 해시만**(평문 저장 금지).
   - `issue()`: 발급 전 활성 상한(5) 강제 — 초과 시 오래된 것부터 폐기(활성 5 유지). 만료 = now + 180일.
   - `rotate()`: 해시로 조회 → 미존재 401 `REFRESH_INVALID` / **이미 폐기된 토큰 재사용 → 프로필 전체 폐기(`revokeAllForProfile`) + 401 `REFRESH_REUSED`**(탈취 방어) / 만료 → 폐기 + 401 `REFRESH_EXPIRED` / 유효 → 기존 폐기+`lastUsedAt` 기록 후 **새 토큰 발급(만료 180일 재연장, 슬라이딩)**.
   - `revoke(profileId, token)`: 해당 리프레시만 폐기(다른 기기 세션 유지, 멱등).
   - `cleanup()`: 만료 + 30일 이상 폐기된 토큰 삭제(정리 크론용).
3. **액세스 토큰 30분 단축** — `AuthService.signToken` 이 `expiresIn`(env `ACCESS_TOKEN_TTL`, 기본 `30m`)으로 서명. JwtModule 전역 기본(7d)은 유지 → **기존 7일 토큰은 만료까지 유효(가드 무변경, 하위 호환)**.
4. **동시 발급** — `verifyPhoneCode`·`kakaoLogin` 성공 시 액세스 + 리프레시 동시 발급, 응답에 `refreshToken` 필드 **additive**. 선택적 `deviceId` 수용(리프레시에 기록).
5. **엔드포인트** (`auth.controller.ts`)
   - `POST /auth/refresh` **(@Public)**: `{ refreshToken, deviceId? }` → 회전 결과 `{ accessToken, refreshToken }`.
   - `POST /auth/logout` (인증): `{ refreshToken }` → `{ revoked }`.
6. **정리 크론** — `RefreshTokenScheduler` `@Cron(EVERY_WEEK, Asia/Seoul)`(일 03:00) → `cleanup()`.

### 앱 (Flutter)
1. **`token_store.dart`** — 리프레시 토큰 secure storage 별도 키(`workon_refresh`) 보관. `writeTokens(access, refresh)`, `readRefresh()`, `clear()`(액세스+리프레시 모두 삭제).
2. **`api_client.dart`** — dio 인터셉터 `onResponse` 에서 401 수신 시:
   - `_refreshOnce()` **single-flight(뮤텍스)**: 진행 중 리프레시가 있으면 그 Future 에 **합류**(동시 다중 401 → refresh 1회). 성공 시 새 토큰 저장 후 **원 요청 1회 재시도**(`_dio.fetch`, `__retried` 플래그로 무한루프 방지). `/auth/*` 경로는 재귀 방지 제외.
   - 리프레시 **실패 시에만** `onUnauthorized`(= 기존 강제 로그아웃 플로우) 호출.
3. **`providers/auth.dart`** — verify/kakao 시 `writeTokens`로 리프레시 저장. **`logout()`이 서버 `POST /auth/logout` 호출(실패 무시) 후 로컬 클리어**. `forceLogout` 은 로컬 클리어(기존 유지).
4. **모델** — `AuthResult` 에 `refreshToken` 추가(nullable, additive).
5. **ARB** — **신규 사용자 문구 불필요**(자동 refresh 는 무음, 세션 완전 만료 시 기존 forceLogout 플로우 재사용). 6개 ARB 무변경.

## 미완 / 관찰 (백로그)
- 앱 `deviceId` 는 현재 미주입(백엔드는 선택 수용, 상한 기능은 동작). 기기별 세션 관리 UI 신설 시 안정적 deviceId 부착 권장.
- `/auth/logout` 은 인증 필요 → 액세스 완전 만료 상태에서 로그아웃 시 인터셉터가 자동 refresh 후 재시도(정상). 리프레시까지 만료면 서버 호출 실패하나 로컬 클리어는 항상 수행.
- 정리 크론은 주 1회. 대량 폐기 누적 방지용이며 기능 정합과 무관.

## 사용자 테스트 스택 반영 방법
- 스키마 변경(마이그레이션 1개, additive) 포함 → 사용자 테스트 스택(DB 5439/백엔드 3010)에 반영하려면 **① 해당 DB에 `prisma migrate deploy` ② 백엔드 재빌드(`npm run build`) 후 재시작**이 필요.
- 신규 액세스 30분 정책은 재시작 후 발급분부터 적용. **기존 발급된 7일 토큰은 그대로 유효**(가드 무변경)하므로 기존 세션은 끊기지 않음.
- 앱은 재빌드 배포 시 리프레시 저장·자동 연장이 활성화(기존 앱은 액세스 만료 시 종전처럼 재로그인 — 하위 호환).
