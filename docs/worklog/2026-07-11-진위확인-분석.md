# SKEP 서류 진위확인 구현 분석 → 작업온 이식 방침 (2026-07-11)

## SKEP의 실제 구조
- Mock 경로(govapi-service/ocr-service Node): 가짜 응답, 폐기됨 → 참고 가치 없음
- **실연동 경로**: document-service → 외부 중개 서버 "lifton"(sk.on1.kr, X-API-KEY 헤더) → 국세청/경찰청RIMS/교통안전공단/KOSHA + Google Vision OCR
- 즉 SKEP도 정부 API를 직접 붙인 게 아니라 중개 서버에 위임

## 제공기관별 API
| 서류 | 실제 제공처 | 발급 난이도 |
|---|---|---|
| 사업자등록 진위/상태 | 공공데이터포털(국세청) — 즉시 발급 | 하 |
| OCR(서류 인식) | Google Cloud Vision | 중 |
| 화물운송 자격 | 교통안전공단(data.go.kr) | 중 |
| 운전/조종사 면허(RIMS) | 경찰청 — 기관 협약 필요 | 상 |
| KOSHA 수료증 | 표준 API 없음, QR+OCR | 상 |

## 작업온 구현 방침 (결정)
1. **1순위: 사업자등록 진위확인** — data.go.kr 서비스키로 직접 연동 (S2에서 구현, 키는 사용자가 발급)
2. OCR·화물은 후순위, **면허/KOSHA는 초기엔 "수동 확인(만료일 등록+서류 열람)" fallback** — SKEP도 UNSUPPORTED 시 수동검증으로 우회함
3. lifton 중개 서버 재사용 여부는 사용자가 접근 권한 보유 시 옵션 (환경변수 VERIFY_* 계열)
4. 이식 가치 높은 SKEP 코드: 서류명→타입→엔드포인트 매핑(VerificationService.java:115), OCR 필드 정규화(collectVerifyFields), 업로드 커밋 후 비동기 검증 패턴, 관리자 수동검증 fallback(markVerified)
5. 마스킹 참고: SKEP은 OCR 응답의 maskedImageBase64로 원본 덮어쓰기 — 작업온은 원본 보존 + 마스킹본 별도 저장(masked_file_path)으로 개선
