## 무엇을 / 왜 / 어떻게

### 백엔드 (완료·검증)
1. **스키마(마이그레이션 1개, 순수 additive)** `20260711053017_p2a_teams`:
   - `teams`(ownerId=반장 profileId, name), `team_members`(teamId FK, profileId? 가입자연결, name, phone?, defaultRate?).
   - `Confirmation`에 `teamId String?` + `teamEntries Json?`([{name, profileId?, quantity(공수), rate, amount}]).
   - `LedgerEntry`에 `sourceConfirmationId String?`(팀 파생 원 확인서 참조, 비-unique) + `derived Boolean`(읽기전용 파생 표시). 기존 `confirmationId @unique`(1:1)는 반장 합계용으로 유지.
   - Confirmation↔LedgerEntry 관계에 이름 부여(`LedgerConfirmation`) + 파생 관계(`LedgerSourceConfirmation`) 추가.
2. **teams 모듈**(`src/teams/`): `/teams` CRUD + 팀원 CRUD(반장 본인만, `ownedTeamOrThrow`). 가입 연결은 전화검색 동의자만(`CONSENT_REQUIRED`), 중복 방지(`TEAM_MEMBER_EXISTS`), 수기는 name 필수. `team-entries.util.ts`(순수 함수) — 팀원별 공수×단가·합계, 공수 0.1 검증(`validateGongsuQuantity` 재사용), 미지정 rate→defaultRate 대체.
3. **팀 확인서**(`confirmations.service.create`): `teamId` 있으면 팀 분기 — 팀 소유 검증 후 `computeTeamEntries`로 서버 계산, `rateType=GONGSU`, `amountCalc.total`=팀 합계, teamEntries 저장, **반장 장부에 합계 1건**. `rateType/rate/quantity`는 팀 모드에서 선택(DTO `@ValidateIf(!teamId)`).
4. **팀원 파생 장부**(`applySignature` → `createTeamDerivedLedger`): 팀 확인서 SIGNED 전이 성공 직후, `teamEntries` 중 가입+연결(profileId 有, 반장 자신 제외) 팀원에게 각자 몫 `ledger_entry`(derived=true, sourceConfirmationId, counterpartyName=반장명) 생성 + 알림. 멱등(같은 원확인서·팀원 중복 skip), 실패는 로그만(서명 차단 안 함).
5. **파생 읽기전용**: `ledger.updateLedger`는 derived면 409 `LEDGER_DERIVED_READONLY`. `addPayment`(입금)는 허용. `entriesForMonth`/`effectiveDate`가 파생 항목을 원 확인서 작업일로 월 집계, `gongsuOf`가 팀 합계/파생 몫 공수 반영. 장부 DTO에 `derived`·`sourceConfirmationId` 노출.
6. **PDF**(`pdf.service.renderConfirmationPdf`): teamEntries 있으면 **팀 명단 표**(이름/공수/단가/금액) + 팀 합계 렌더.
7. **공개 API**(`publicView`): `teamEntries`·`isTeam` 포함(웹 PaperConfirmation P2 통합용).

## 미완 / 다음 배치 주의사항
- **웹 이월(P2 통합)**: 공개 API(`GET /public/confirmations/:token`)에 `teamEntries`·`isTeam` 이미 포함. 웹 `PaperConfirmation`에 팀 명단 표 렌더는 **P2 통합 배치에서** 처리(이번엔 웹 수정 금지 준수).
- **P2b(표준근로계약서) 주의**: 서명 인프라(`applySignature`·`shareToken`·공개 서명)를 재사용할 것. 팀 파생처럼 "서명 시점 후속 처리"가 필요하면 `applySignature` 말미(SIGNED 전이 성공 후)에 훅을 추가하는 패턴을 따를 것. LedgerEntry에 `derived`/`sourceConfirmationId`가 생겼으니 계약서-장부 연동 시 confirmationId unique 제약(1:1) 유의.
- 팀 DRAFT 확인서의 `PATCH /confirmations/:id` 부분수정은 팀원 금액 재계산을 지원하지 않음(생성·재작성 흐름). 필요 시 별도 팀 편집 엔드포인트 추가 여지.
