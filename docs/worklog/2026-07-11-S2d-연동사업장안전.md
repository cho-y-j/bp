# S2d 연동·작업지시·사업장·안전·알림 (2026-07-11)

## 구현 범위
S2a~S2c 위에 백엔드 마지막 배치(S2d)를 구현했다. 기존 공통 서비스(FileStorageService/PdfService/NotificationsService/dday·time.util/ConfirmationsService)를 **재사용**하고 중복 구현하지 않았다.

### 스키마 (마이그레이션 `20260711040000_s2d_connections_safety`)
- Business: `address?`, `lat?`, `lng?`(폭염 좌표 조회) 추가.
- SafetyLog: `ackAt?` 추가 — append-only 예외로 **최초 1회만 UPDATE** 허용(재확인 409).
- 신규 테이블 없음(Business/Connection/Job/WorkLog/SafetyLog/Notification/DeviceToken 본체는 S1 스키마에 존재).
- ⚠️ S2b/S2c 와 동일: 머신 시계 기준 타임스탬프가 앞서 정렬되는 문제 → 폴더명을 `20260711040000_...` 로 지어 deploy 순서(init→s2a→s2b→s2c→s2d) 보장. fresh `migrate deploy` 로 순서 검증 완료.

### 연동 `/businesses`, `/workers`, `/connections` (connections 모듈)
- `POST /businesses`: 6자리 숫자 초대코드 자동 발급(중복 회피 재시도) + 생성 즉시 **미가입 상대 승격** 실행.
- `GET /businesses/search?q=`: 코드(6자리) 정확일치 or 상호 부분일치(대소문자 무시). 코드매칭 항목만 `matchedByCode` 표시.
- `GET/PATCH /businesses/mine`: 소유 사업장 목록/수정(다중 소유 시 body.id 로 대상 지정).
- `GET /workers/search?phone=`: **phoneSearchConsent=true 프로필만** 노출, 이름 마스킹(`홍길동`→`홍*동`, `common/phone.util.maskName`).
- `POST /connections`: 방향 자동 판별 — workerProfileId 있으면 사업장→작업자(요청자=소유자 검증), 없으면 작업자→사업장(요청자=작업자). `@@unique([profileId,businessId])` 중복 409. 수신자에게 알림.
- `POST /connections/:id/accept`: 작업자·소유자만 수락(409 재수락), **수락 시 승격 재실행**. `GET /connections`(양측·역할표시), `DELETE /:id`.
- **미가입 상대 승격**(`PromotionService.promoteForBusiness`): 사업주 프로필 전화 매칭으로 businessId=null 인 확인서·작업지시(manualContact)를 승격, 승격된 확인서에 연결된 **ledger_entry 도 대칭 갱신**(businessId 채우고 counterpartyName 제거). 매칭은 순수 함수(`promotion.util.selectPromotable`, 전화 정규화 후 비교) — 단위 테스트.

### 작업지시 `/jobs` (jobs 모듈)
- `POST /jobs`: 소유 사업장 + **ACCEPTED 연결 작업자**에게만 지시/예약(Job SCHEDULED) → 작업자 RESERVATION 알림.
- `GET /jobs?month=`: 양측(작업자/소유자), scheduledAt 월 필터, 역할·최근 WorkLog 요약 포함.
- `POST /:id/confirm`: 작업자 수락(acceptedAt) + **서류 유효성 자동 확인**(작업자+장비 서류 중 만료 지난 것 스캔) → `safety_log(DOCUMENT_VALIDITY, 결과 payload)` 기록, 만료 있으면 사업장에 알림.
- `POST /:id/start`: GPS + 컨디션체크(OK|BAD) → `work_log`(startedAt/gps/conditionCheck) + `safety_log(CONDITION_CHECK)`. BAD 면 사업장 알림. status=IN_PROGRESS.
- `POST /:id/complete`: GPS + 사진경로 → work_log 갱신, status=DONE, 사업장 완료 알림.
- `POST /:id/photos`(multipart): **FileStorageService 재사용**, `uploads/{profileId}/{jobId}/job-photo-*` 저장 → work_log.photoPaths 병합.

### 사업장 `/biz` (biz 모듈)
- `GET /biz/inbox`: 소유 사업장 대상 수신 확인서(SENT/SIGNED) 목록·상태(작업자명 마스킹).
- `POST /biz/confirmations/:id/sign`: 앱 내 서명 — **ConfirmationsService.applySignature 공통화 후 재사용**(bizSign), 권한은 해당 사업장 소유자. public sign 과 동일 PNG 검증·SIGNED 전이·발행자 알림·재서명 409.
- `GET /biz/settlements?month=`: **SIGNED 확인서 기준** 작업자별 미지급 집계(ledger 미수와 대칭), 미수 항목의 ledgerEntryIds 반환.
- `POST /biz/settlements/pay`: `{ ledgerEntryIds, paidAt?, memo? }` → 각 ledger 에 **미수 전액을 입금 기록(byBusiness)** + PAID 전이. 작업자 장부와 **동일 레코드/데이터**(양측 일치) + 작업자 입금 알림. 완납 항목은 중복 입금 안 함.
- `GET /biz/safety-report?month=`: safety_logs(businessId) 집계 PDF — 유형별 건수 + 발송/확인 표(일자·유형·대상·확인시각). **PdfService.renderSafetyReportPdf** 신규(한글 폰트 재사용).

### 안전 `/safety` (safety 모듈) — 폭염 자동 알림
- `WeatherService`: 기상청 단기예보(`KMA_SERVICE_KEY`) — 위경도→격자 변환(`latLngToGrid`), TMX/TMP 최고기온 파싱. **키 없으면 비활성(로그만, getReading→null)**.
- `HeatwaveScheduler`(모두 **Asia/Seoul**): 06:00 활성 연결 사업장(lat/lng+ACCEPTED) 폭염 스캔 → 33°C↑ 이면 연결 작업자 전원+사업주에게 경고 알림 + `safety_log(HEAT_ALERT, businessId)`. 14:00 같은 날 폭염 감지 사업장에 휴식 안내(REST_GUIDE). 같은 날 중복 방지.
- 폭염 판정은 순수 함수(`heatwave.util.isHeatwave`, 최고기온/체감온도 max ≥ 임계) — 단위 테스트.
- `POST /safety/:logId/ack`: 대상 작업자 본인만, `ackAt` 최초 1회 UPDATE(이미 있으면 **409**).
- `POST /safety/simulate-heatwave`: **dev 전용**(NODE_ENV=production 이면 403, @Public 아님) — 기상청 없이 소유 사업장에 폭염 플로우(35°C simulated) 트리거. 알림 data 에 `safetyLogId` 실어 ack 딥링크 지원.

### 알림 — FCM 실발송 (notifications 모듈 보강)
- `FcmService`: `FCM_SERVICE_ACCOUNT_PATH` 로 firebase-admin **지연 로드** 초기화. **키 없으면 비활성(로그만)** — 현재 키 미보유에서도 부팅/발송 안전. 무효 토큰(등록취소/invalid) 반환.
- `NotificationsService.create`: 레코드 생성 후 device_tokens 조회 → FCM 발송 시도(성공/실패 로그) → 무효 토큰 정리. 실패해도 예외 안 던짐.
- `GET /notifications?unread=`, `POST /notifications/:id/read`, `POST /device-tokens`(token upsert, 전역 unique) 신규 컨트롤러.
- 크론 TZ: 스케줄 전체 `Asia/Seoul` 명시 — 기존 `document-expiry`·`ledger-due`(EVERY_DAY_AT_9AM) 포함 일괄 정리.
