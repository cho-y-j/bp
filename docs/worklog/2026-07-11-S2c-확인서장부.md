# S2c 확인서·장부 도메인 (2026-07-11)

## 구현 범위
S2a(인증)·S2b(서류·장비) 위에 작업확인서·장부 도메인을 구현했다. 기존 공통 서비스(FileStorageService/PdfService/dday.util/NotificationsService)를 **재사용**하고 중복 구현하지 않았다.

### 스키마 (마이그레이션 `20260711030000_s2c_confirmations_ledger`)
- Confirmation 보강(AlterTable 3컬럼): `manualContact`(수기 상대 연락처), `revokedAt`(링크 무효화), `viewLogs Json[]`(공개 열람 로그).
- Confirmation/LedgerEntry 본체 모델은 S1 스키마에 이미 존재 → 신규 테이블 없음.
- ⚠️ S2b 와 동일 이슈: 머신 시계 기준 생성 타임스탬프(`20260710172247`)가 init/s2a/s2b 보다 앞서 정렬됨 → 폴더명을 `20260711030000_...` 로 **수동 리네임**해 deploy 순서(init→s2a→s2b→s2c) 보장. fresh `migrate deploy` 로 순서 검증 완료.

### /confirmations (작성자 본인만)
- `POST /confirmations`: 코어(date, siteName, businessId? 또는 수기 companyName/contact, workDescription, startTime~endTime, rateType DAILY|HOURLY|PER_CASE + rate + quantity + additionalItems[연장/조출/야간/철야/기타]) + equipmentSection JSONB.
  - **amountCalc 서버 계산**(`amount.util.ts`, 순수 함수): 항목별 금액(단가×수량) + 공급가 합계 + 부가세(선택 vatRate) + 청구 합계. SKEP 단가 5종(기본/연장/조출/야간/철야) 차용. 클라이언트 금액 불신 — 항상 재계산.
  - **같은 트랜잭션으로 ledger_entry 자동 생성**(`$transaction`): 금액=청구합계, 수금예정일=body.dueDate(기본 null), counterpartyName=수기명(연동 시 null).
  - shareToken 은 **생성 시 nanoid(32) 발급**(스키마상 non-null·unique).
- `POST /:id/duplicate`: 날짜만 오늘(KST)로, 시각은 원본 HH:mm 을 오늘 날짜에 재조합, DRAFT + 새 토큰 + 새 ledger.
- `GET ?month=YYYY-MM`: 목록 + **일자별 집계**(byDate: date/count/totalAmount) + 월 합계 — 캘린더 월/주 뷰용.
- `GET /:id`, `PATCH /:id`(**DRAFT 만**, 금액 변경 시 amountCalc 재계산 + ledger 금액 동기화, 부분 수정은 기존 amountCalc 항목 복원), `DELETE /:id`(**DRAFT 만**, 연결 ledger 도 트랜잭션으로 제거).
- `POST /:id/send`: DRAFT→SENT. 연동 사업장이면 사업장 owner 에게 CONFIRMATION 알림 레코드 생성, 수기 상대면 링크만. **두 경우 모두** `{ shareToken, url: PUBLIC_WEB_URL/c/{token}, linked, notified }` 반환.
- `GET /:id/pdf`: 작업확인서 PDF(제목 "작업확인서", 필드 표, 장비 섹션, 금액 표, 특이사항, 서명란). **한글 폰트 로컬 임베드**(fontkit + subset).
- `GET /public/confirmations/:token` (@Public): 열람(만료 없음, `revokedAt` 무효화만 403 체크) + viewLog append. `POST /public/confirmations/:token/pdf` 는 서명 반영본.
- `POST /public/confirmations/:token/sign` (@Public): `{ signerName, signImageBase64(PNG data URI ≤1MB) }` → PNG 시그니처/크기 검증 후 파일 저장, signedAt(서버시각)·status=SIGNED, 발행자에게 알림. **이미 SIGNED 면 409 ALREADY_SIGNED**. 이후 PDF 에 서명 이미지 삽입.

### /ledger (본인 장부)
- `GET /ledger/summary?month=`: 일한 날 수(distinct KST 작업일) / 총 청구액 / 미수 합계 / 입금 합계.
- `GET /ledger/by-company?month=`: 상대별(연동 businessId or 수기명) 그룹 — 일수/합계/입금/미수, 수금예정 최근일 D-day, 상태(전액입금/부분입금/미수/기한지남).
- `POST /ledger/:id/payments`: 부분입금 `{ amount, paidAt?, memo? }` → payments JSONB append, 합계≥금액이면 PAID. `PATCH /ledger/:id`: dueDate 수정(+상태 재계산).
- `GET /ledger/statement?month=`: 월간 명세서 PDF(상대별 소계 + 총계 표, 페이지 넘침 시 자동 개행).
- 크론 `LedgerDueScheduler`(매일 09:00, 기존 서류 만료 크론과 동일 시간대에 추가): 미수 장부 중 수금예정 D-1/D-0 → PAYMENT_DUE 알림. 같은 장부·D-day 중복 방지.

## S2d(연동·사업장·안전·알림) 이월 주의
- **미가입 상대 승격**: 현재 확인서/장부의 수기 상대(companyName/manualContact, counterpartyName)는 businessId=null. 사업장 가입 시 전화번호 매칭으로 연결 승격 로직 필요(DESIGN §5). `manualContact` 에 연락처 저장돼 있음 → 매칭 키로 사용.
- **send 알림 발송(FCM)**: 현재 send/sign 은 Notification 레코드 + sendPush 로그 스텁만. S2d 에서 FCM 실발송으로 대체(device_tokens 조회). 수금 D-day 크론(`LedgerDueScheduler`)도 동일.
- **사업장 수신함/앱내 서명**(API-SPEC `/biz/inbox`, `/biz/confirmations/:id/sign`): 연동 상대가 앱에서 서명하는 경로는 미구현. 현재는 shareToken 공개 링크 서명만(연동이어도 링크 공유 가능). biz 서명은 confirmation 을 businessId 소유자 관점에서 조회·서명하는 별도 컨트롤러 필요.
- **정산(`/biz/settlements`)**: 사업장이 작업자에게 지급하는 방향은 미구현. 현재 ledger 는 작업자가 **받을 돈** 관점(단방향). 사업장 지급 처리 시 해당 ledger 에 payment append 하는 연동 지점 고려.
- **크론 시간대**: `EVERY_DAY_AT_9AM` 은 서버 로컬 TZ 기준. 운영 컨테이너 TZ 를 KST(Asia/Seoul)로 고정해야 09:00 KST 발송. D-day 계산 자체는 KST 고정이라 안전하나 발송 시각은 TZ 의존.
- **폰트 라이선스**: 로컬 AppleGothic 복사본은 개발/검증용. 운영 배포 시 배포 가능 라이선스 폰트(예: 나눔고딕/Pretendard OFL)로 교체 권장.
- **부가세**: amountCalc 에 vatRate 지원(기본 0). 세금계산서 자동발행(Phase 3)과 연계 시 이 값 활용.
- **RateType**: API 계약의 PER_JOB(건당)은 스키마 enum `PER_CASE` 로 매핑. MONTHLY/UNIT 은 확인서 DTO 에서 미허용(DAILY/HOURLY/PER_CASE 만) — 필요 시 확장.
- 마이그레이션 타임스탬프 리네임 주의(머신 시계 이슈) — S2b/S2c 동일 패턴, deploy 순서 폴더명으로 강제.
- 로컬 dev DB 는 5433(임시)·5432 는 live-db(타 프로젝트) — **절대 5432 사용 금지**.
