1. **스키마 변경 없음** — 기존 `ledger_entries`(payments/dueDate/derived/sourceConfirmationId) + `confirmations`(teamEntries) 재사용. 마이그레이션 추가 없음.
2. **집계 순수 함수** `src/ledger/income-report.util.ts` (`aggregateIncomeReport(rows, months)`):
   - 월별 추이(청구/입금/미수·일한 날·공수) — `months` 로 전체 월을 0으로 채워 추이 그래프용 배열 생성.
   - 상대별(회사명·건수·총액·입금·미수, 총액 내림차순).
   - 총계(총 청구·입금·미수·일수(기간 내 distinct 날짜)·공수·건수·**teamPayout·netBilled**).
   - **파생 중복 합산 방지 검증(내장)**: ①파생 항목(팀원 몫, `derived=true`)에 `teamPayout≠0` 이면 예외, ②최종 정합성(월별 청구합 = 상대별 총액합 = totalBilled) 불일치 시 예외.
   - `incomeTaxNoticeKo(period)` — 종소세 일반 안내 5줄(5월 신고·인적용역 3.3% 원천징수·정산·경비 보관·**세무 상담 아님** 명시).
3. **서비스** `LedgerService.incomeReport / incomeReportPdf`:
   - `resolveRange` — `year`(YYYY→1~12월) 또는 `from&to`(YYYY-MM, 분기 등). `enumerateMonths` 로 월 목록·최대 24개월 검증(초과 400 RANGE_TOO_LONG, 역순 400 INVALID_RANGE, 형식 오류 400 INVALID_YEAR/INVALID_MONTH, 누락 400 INVALID_RANGE).
   - `entriesForRange` — 월간 조회(`entriesForMonth`)와 동일 기준(확인서 작업일 우선·팀 파생 원 확인서 작업일·수기 생성일)으로 범위만 [start,end) 확장.
   - `normalizeIncomeRows` — 각 항목을 `{month, workDate, amount, paid(computeOutstanding), outstanding, gongsu(gongsuOf), businessId, companyName, teamPayout(teamPayoutOf), derived}` 로 정규화.
   - **`teamPayoutOf(e, userId)`**: 팀 확인서(teamEntries 有·비-파생)에서 팀원(profileId≠본인) 몫 amount 합. 파생/일반 항목은 0. → `netBilled = totalBilled − teamPayout`(반장 본인 몫 = 순소득 참고, 차감 아님).
4. **팀 파생 처리 원칙**(현행 장부와 동일): 팀원 파생 항목(`profileId=팀원`)은 팀원 리포트에서 본인 소득으로 자연 집계(teamPayout 0). 반장 리포트는 팀 확인서 전체(합계 1건)가 매출로 잡히고, 팀원 지급분을 `teamPayout` 으로만 별도 표기. 한 사용자 리포트 내에서는 파생 항목과 팀 합계 항목이 공존하지 않으므로(반장 자신은 파생 제외) 이중 합산 없음 — 순수 함수 검증으로 회귀 방지.
5. **PDF** `PdfService.renderIncomeReportPdf` — 총계 요약 + 월별 추이 표 + 상대별 표 + 종소세 안내(나눔고딕·A4·페이지 브레이크, `renderStatementPdf`/`renderSafetyReportPdf` 패턴 재사용).
6. **컨트롤러** `GET /ledger/income-report`(JSON) · `GET /ledger/income-report/pdf`(인증 blob, `Content-Disposition inline`). 기존 JWT 가드 하 — 미인증 401.

## 미완 / 다음 배치(P2 통합 검수) 주의사항
- **teamPayout 정의**: "팀원 몫"은 팀 확인서 teamEntries 중 profileId≠반장 전체(가입 파생 + 수기 미가입 포함). 즉 파생 ledger(가입자만)보다 넓게 잡아 순소득 참고의 정확도를 높임. 반장 본인이 teamEntries 에 포함되면 그 몫은 netBilled 에 남음.
- **curl 시드 vs e2e**: 수동 검증 시 curl 시드는 입금 미기록·상대 구성이 e2e와 달라 숫자가 다름(둘 다 자체 정합). P2 통합 시 실데이터로 재확인 권장.
- **income-02/03 스크린샷 동일 지점**: 리포트 하단이 한 화면에 들어와 두 스크린샷이 사실상 같은 영역(상대별+종소세+PDF). 총계·막대는 income-01 에서 확인.
- **P2 통합 검수 포인트**: ①팀 반장/팀원 양쪽 리포트 합산이 팀 전체에서 이중 계산되지 않는지(반장 매출 = 팀 전체, 팀원 소득 = 파생 몫 — 팀 단위 합산 시 반장+팀원 단순합은 중복이므로 통합 리포트에선 주의). ②세금계산서 발행분(taxInvoicedAt) 제외는 소득 리포트에서 미적용(소득 집계는 발행 여부와 무관 — 필요 시 정책 확정). ③웹 소득 리포트 UI 없음(앱 전용, 필요 시 통합에서).
