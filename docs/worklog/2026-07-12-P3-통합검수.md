## 3. 코드 리뷰 스팟
- **크론 3개 시간 충돌**: 배지 04:00 · (기존)수금예정 09:00 · 독촉 10:00 KST. 09:00 은 문서만료 크론과 슬롯 공유하나 **서로 다른 테이블·멱등 dedup** 이라 트랜잭션/락 충돌 없음. 정상.
- **cardToken 엔트로피**: `nanoid(32)`(64자 알파벳 ≈ 192bit). 무차별 대입 비현실적. 확인서 `shareToken`(길이 32, 라이브 실측 `K4bq6rSaoBhtVf3xLv-…`)·문서 share(32)와 **길이 일관** ✓
- **viewCount 동시성**: `profile.update({ cardViewCount: { increment: 1 } })` — Prisma 원자적 증가. read-modify-write 경합 없음 ✓
- **배지 캐시 무효화 타이밍**: pay 커밋 이후(메서드 말미) `void Promise.all(recomputeBusinessQuietly)` fire-and-forget → 커밋된 데이터 재집계, 실패해도 pay 응답 무영향 + 일일 크론(04:00) 최종 정합. 정상.
- **마이그레이션**: P3a/P3b 둘 다 **additive-only**(ADD COLUMN / CREATE UNIQUE INDEX, DROP/NOT-NULL-add 없음). 기존 데이터 무손상.

## 4. 관찰 사항 (LOW — 출시 차단 아님, 백로그 권고)
1. **[LOW] `autoReminderStage` 정확 일치 판정**(`reminder.util.ts:33`): `dday === -7`/`=== -30` 정확 일치. 크론이 하루라도 건너뛰거나(서버 다운) 작업자가 D+7 경과 후 autoRemind 를 켜면 해당 단계는 **영구 미발송**(dedup 은 정상 작동). "D+7/D+30 도달" 스펙엔 부합하나, `dday <= -7`(미발송 단계만) 로 완화하면 늦게 켠 사용자도 놓치지 않음. 설계 트레이드오프.
2. **[LOW] `reminders` Json[] 상한 부재**: 자동은 D7+D30(단계 dedup 로 2건)로 사실상 유계, 수동은 3일 쿨다운으로 제한. 다만 장기 미수 항목에 수동 발송을 수년 반복하면 배열이 계속 증가. 최근 N건 cap(예: 50) 권고. (worklog P3a 도 "cap 필요?" 로 자문)
