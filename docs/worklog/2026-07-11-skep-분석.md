# SKEP 레퍼런스 프로젝트 분석 (2026-07-11)

- 소스: https://github.com/cho-y-j/skep.git → `ref/skep`에 클론 (참고 전용, 우리 코드 아님)
- 분석: Opus 서브에이전트 수행

## 정체
SK 대형 건설현장용 "장비 투입 관리 플랫폼". 시행사→BP사→공급사→점검원→기사 5단계 공급망 전체를 모델링.

## 기술 스택
- Flutter Web + Java 21/Spring Boot 3.2 **마이크로서비스 11개** + PostgreSQL 16(PostGIS) + Redis
- 총 125개 API 중 프론트 연동은 25개뿐 (오버빌드)

## 우리가 가져올 것 (Good)
1. **작업확인서 필드 세트** (`daily_work_confirmations`): 현장명, 업체명, 차량번호, 기사명, 장비명, 작업내용, 작업위치, 규격, 시작/종료시각, 초과/철야시간, 서명자+서명시각, 상태(PENDING_SIGNATURE→SIGNED)
2. **정산 로직**: 단가 5종(기본/연장/조출/야간/철야) 분해 + 일자별 detail 합산 + 공급가+부가세10% 자동계산, 상태 DRAFT→SENT→PAID
3. **서류 만료 관리 패턴**: 서류유형에 `has_expiry` 플래그 + `expiry_date` 인덱스 + "만료 임박 N일" 조회 API
4. **엔티티별 필수서류 매핑** 테이블 패턴

## 우리가 버릴 것 (Avoid)
- 11개 MSA/서비스별 DB → **단일 백엔드 + 단일 DB**
- 7개 역할 RBAC, 견적 승인 라인, 투입 체크리스트 7게이트 → **작업자↔사업장 2자 관계**로 단순화
- 일일/월간 확인서 이원화 → **단일 확인서** + 집계 파생
- NFC, 지문, 지오펜싱, 실시간 위치추적(WebSocket), OnlyOffice, 정부API 4종 → 제거. 증빙은 사진 + GPS 스냅샷 + 터치 서명이면 충분
- 화면 없는 API 만들기 금지

## SKEP의 약점 = 우리의 차별점
- SKEP의 "서명"은 서명자 UUID+시각만 기록 → 우리는 **실제 터치 서명 이미지** 저장
- SKEP은 양쪽 모두 가입 전제 → 우리는 **미가입 상대도 웹 링크로 확인·서명** 가능 구조
